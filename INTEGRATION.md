# Integration Points — xGE Internal Comms Triage Agent

**Version:** 1.0  
**Last Updated:** February 2026  
**Platform:** Google Workspace + Gemini API

---

## Architecture Overview

The Comms Triage Agent is not a standalone tool — it is an orchestration layer connecting six distinct systems. Understanding how data flows between them is essential for debugging, extending, and maintaining the agent.

```
┌─────────────────────────────────────────────────────────────────┐
│                    INTEGRATION MAP                               │
│                                                                  │
│  [Google Form]  ──trigger──▶  [Apps Script]                     │
│                                     │                            │
│                             ┌───────┴────────┐                  │
│                             ▼                ▼                   │
│                      [Google Sheets]   [Gemini API]             │
│                      (intake + log)   (3-prompt chain)          │
│                                             │                    │
│                             ┌───────────────┤                   │
│                             ▼               ▼                   │
│                         [Gmail]       [Google Chat]             │
│                      (requesters)    (escalations)              │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## Integration 1: Google Forms → Apps Script

**Role:** Intake — this is where all requests enter the system.

### How It Works

When a requester submits the xGE Comms intake form, Google Forms automatically appends a new row to a linked Google Sheet. This triggers the Apps Script `onFormSubmit` event handler, which is the entry point for all agent logic.

### Data Flow

```
Requester fills form
        ↓
Form appends row to linked Sheet
        ↓
onFormSubmit(e) fires in Apps Script
        ↓
e.values[] array contains all field responses
```

### Field Mapping

The agent reads form responses by column index from `e.values[]`. **Column order must match form question order exactly.** If the form is reordered, update the index map in `Config.gs`:

```javascript
var FIELD_INDEX = {
  TIMESTAMP:            0,
  REQUESTER_EMAIL:      1,
  TEAM:                 2,
  REQUEST_TYPE:         3,
  CONTENT_STATUS:       4,
  SUBJECT:              5,
  SUMMARY:              6,
  SCOPE:                7,
  AUDIENCE_DETAIL:      8,
  LEADERSHIP_VISIBILITY:9,
  URGENCY:              10,
  SENDER:               11,
  DRAFT_LINKS:          12,
  ADDITIONAL_NOTES:     13
};
```

> **Critical dependency:** If a form field is added, removed, or reordered, this map must be updated before the next deployment. Misaligned indices are the most common source of silent data errors.

### Form URL

The intake form URL is stored in `CONFIG.INTAKE_FORM_URL` for reference but is not used programmatically — the trigger fires on submission automatically.

## Integration 2: Google Sheets — Intake + Logging

**Role:** Persistence — the source of truth for all request data and agent decisions.

### Sheet Structure

The linked Google Sheet has two tabs:

**Tab 1: Form Responses 1** (auto-generated by Google Forms)
- Read-only from the agent's perspective
- Source of all intake data
- Do not modify this tab manually — it breaks the form linkage

**Tab 2: Agent Log** (managed by agent)
- Written to by the agent after every processed request
- Used for auditing, debugging, and reporting
- Columns: Timestamp, Requester, Request Type, Touch Level, Confidence Score, Agent Action, Status, Notes

### How the Agent Writes to the Log

```javascript
function logToSheet(data) {
  var sheet = SpreadsheetApp.openById(CONFIG.RESPONSE_SHEET_ID)
                            .getSheetByName(CONFIG.LOG_SHEET_TAB);
  sheet.appendRow([
    new Date(),               // Timestamp
    data.requesterEmail,      // Requester
    data.requestType,         // Request Type
    data.touchLevel,          // Touch Level
    data.confidenceScore,     // Confidence Score
    data.agentAction,         // Agent Action
    data.status,              // Status
    data.notes                // Notes
  ]);
}
```

### Log Status Values

| Status | Meaning |
|--------|---------|
| `COMPLETED` | Low Touch — agent handled autonomously, output delivered |
| `ESCALATED` | Medium/High Touch — routed to escalation owner |
| `PENDING_REVIEW` | Agent flagged for human review (confidence 70–89%) |
| `ERROR` | Unexpected failure — see Notes column |
| `TEST` | Dry-run test entry — not a real request |

## Integration 3: Gemini API — The AI Brain

**Role:** Execution — all classification, revision, and escalation reasoning happens here.

### Three-Prompt Architecture

The agent makes up to three sequential Gemini API calls per request:

```
Request
  │
  ▼
[Prompt 1: TRIAGE]
  → Input: Full form data
  → Output: Touch level + confidence score + reasoning
  │
  ├─ Low Touch → [Prompt 2: REVISION]
  │                → Input: Draft content + triage context
  │                → Output: Revised draft + rationale per change
  │
  └─ Medium/High Touch → [Prompt 3: ESCALATION]
                           → Input: Form data + triage reasoning
                           → Output: Escalation summary for [ESCALATION_OWNER]
```

Each prompt call is independent. The output of Prompt 1 is passed as context into Prompts 2 and 3, but they are separate API calls, not a single conversation.

### API Call Structure

```javascript
function callGemini(systemPrompt, userPrompt) {
  var apiKey = PropertiesService.getScriptProperties().getProperty('GEMINI_API_KEY');
  var url = 'https://generativelanguage.googleapis.com/v1beta/models/'
            + CONFIG.GEMINI_MODEL + ':generateContent?key=' + apiKey;

  var payload = {
    system_instruction: { parts: [{ text: systemPrompt }] },
    contents: [{ parts: [{ text: userPrompt }] }]
  };

  var options = {
    method: 'post',
    contentType: 'application/json',
    payload: JSON.stringify(payload),
    muteHttpExceptions: true
  };

  return UrlFetchApp.fetch(url, options);
}
```

### Error Handling

The agent implements exponential backoff for Gemini API rate limit errors (429):

```javascript
function callGeminiWithRetry(systemPrompt, userPrompt) {
  var attempts = 0;
  var delay = CONFIG.RETRY_DELAY_MS;

  while (attempts < CONFIG.MAX_RETRIES) {
    var response = callGemini(systemPrompt, userPrompt);
    var code = response.getResponseCode();

    if (code === 200) {
      return JSON.parse(response.getContentText());
    }

    if (code === 429) {
      Utilities.sleep(delay);
      delay = delay * 2;  // Exponential backoff
      attempts++;
    } else {
      throw new Error('Gemini API error: ' + code + ' — ' + response.getContentText());
    }
  }

  throw new Error('Gemini API: max retries exceeded');
}
```

### Prompt Injection Points

Each prompt has named injection points where the knowledge base is inserted at runtime:

```javascript
// In Triage.gs
var triagePrompt = TRIAGE_SYSTEM_PROMPT
  .replace('[TRIAGE_CRITERIA]',   loadKnowledgeBase('triage_criteria'))
  .replace('[EXAMPLES]',          loadKnowledgeBase('triage_examples'))
  .replace('[HARD_RULES]',        loadKnowledgeBase('hard_rules'));
```

Knowledge base content is stored in the `KnowledgeBase` tab of the Google Sheet and loaded at runtime, so prompt content can be updated without modifying code.

## Integration 4: Gmail — Output Delivery

**Role:** Output — delivering revised content and confirmations to requesters.

### What the Agent Sends via Gmail

| Scenario | Email to Requester | Email to [ESCALATION_OWNER] |
|----------|--------------------|-----------------|
| Low Touch — complete | Revised draft + rationale | None |
| Medium Touch — escalated | "Your request has been received and routed" | Escalation summary |
| High Touch — escalated | "Your request has been received and routed" | Escalation summary |
| Error / unclear | "We need clarification before proceeding" | Optional flag |

### Email Structure — Low Touch Output

```javascript
function sendRevisionEmail(requesterEmail, revisedContent, rationale) {
  GmailApp.sendEmail(
    requesterEmail,
    'xGE Comms: Your revision is ready',
    '',
    {
      htmlBody: buildRevisionEmailHTML(revisedContent, rationale),
      from: CONFIG.SENDER_ALIAS,
      name: 'xGE Communications'
    }
  );
}
```

> **Invisible AI operation:** All emails are sent from `xge-comms@domain.com` with display name "xGE Communications." No AI attribution appears in the email. This is intentional for the AI-skeptical audience.

### Alias Requirements

The sender alias (`CONFIG.SENDER_ALIAS`) must be configured as a "Send mail as" address in the deploying account's Gmail settings before deployment. The account running the script must have permission to send from this alias.

## Integration 5: Google Chat — Escalation Notifications

**Role:** Real-time escalation alerts to the coverage owner ([ESCALATION_OWNER], or designated backup).

### How Chat Notifications Work

Medium and High Touch escalations post a structured message to a designated Google Chat Space via an incoming webhook. No bot account is required — this is a simple HTTPS POST.

```javascript
function notifyEscalationOwner(escalationData) {
  var message = {
    text: buildEscalationMessage(escalationData)
  };

  var options = {
    method: 'post',
    contentType: 'application/json',
    payload: JSON.stringify(message)
  };

  UrlFetchApp.fetch(CONFIG.CHAT_WEBHOOK_URL, options);
}
```

### Escalation Message Format

```
🔴 New Request — HIGH Touch

Requester: [Name, Team]
Request: [One-sentence summary]

Classification Reasoning:
• Scope: Org-wide → High
• Audience: VP+ visibility → High
• Timeline: <3 days → High
• Key factors: Hard rule triggered (leadership name detected)

Confidence: 98%

Recommended Approach: Full partnership required. Schedule discovery call.

Requester notified: Yes
```

### Webhook Setup

1. Open the target Google Chat Space
2. Click the space name → **Apps & Integrations → Webhooks**
3. Click **Add Webhook**, name it `Comms Triage Agent`
4. Copy the webhook URL
5. Store it in `Config.gs` as `CHAT_WEBHOOK_URL`

> **Security note:** Treat the webhook URL as a secret. Anyone with this URL can post to your Chat Space. Do not commit it to a public repository — store it in Script Properties alongside the API key.

## Integration 6: Knowledge Base (Google Sheets Tab)

**Role:** Runtime context — the structured rules, examples, and templates that ground all AI decisions.

### Knowledge Base Tab Structure

The `KnowledgeBase` tab in the Agent Sheet stores named blocks of text that are injected into prompts at runtime:

| Column A | Column B |
|----------|----------|
| `triage_criteria` | Full text of TRIAGE_CRITERIA.md |
| `triage_examples` | Labeled classification examples |
| `hard_rules` | Hard escalation trigger rules |
| `revision_rules` | Smart Brevity and editing guidelines |
| `audience_profiles` | L8+ IC and VP/Director preferences |
| `escalation_template` | Format for [ESCALATION_OWNER] escalation notifications |

### Loading Knowledge Base at Runtime

```javascript
function loadKnowledgeBase(key) {
  var sheet = SpreadsheetApp.openById(CONFIG.RESPONSE_SHEET_ID)
                            .getSheetByName('KnowledgeBase');
  var data = sheet.getDataRange().getValues();

  for (var i = 0; i < data.length; i++) {
    if (data[i][0] === key) {
      return data[i][1];
    }
  }

  throw new Error('Knowledge base key not found: ' + key);
}
```

### Updating Knowledge Base Content

To update a triage rule, revision example, or audience profile:

1. Open the Agent Sheet
2. Navigate to the `KnowledgeBase` tab
3. Find the row with the relevant key (Column A)
4. Edit Column B directly
5. **No code changes or redeployment required** — the agent loads content at runtime

This design principle — separating knowledge from code — is what allows non-engineers to maintain the agent's behavior over time.

## Future Integration Points

These integrations are planned for future releases and are scoped but not yet built:

### Google Drive — Document Retrieval
**Use case:** Low Touch requests that reference a doc by URL. Agent fetches the doc content directly via Drive API rather than requiring a copy-paste.  
**Status:** Scoped — requires Drive API OAuth scope added to Apps Script project.

### Google Forms — Dynamic Response
**Use case:** After triage, agent sends a pre-filled follow-up form to the requester when clarifying questions are needed.  
**Status:** Potential — requires Forms API.

### [INTERNAL_KB]
**Use case:** Agent queries internal org charts and style guides for audience context during high-stakes revisions.  
**Status:** Blocked — [INTERNAL_PROXY] does not allow external API calls to internal [INTERNAL_KB] endpoints. Requires internal tooling solution.

### Slack / External Chat
**Use case:** Port escalation notifications to Slack for organizations not using Google Chat.  
**Status:** Simple — replace webhook URL and message format. Documented in `DEPLOYMENT.md` under Customization.

Use this checklist to verify all integrations are functioning after deployment or a change:

| Integration | Health Check | Expected Result |
|-------------|-------------|-----------------|
| Form → Apps Script | Submit test form entry | `onFormSubmit` fires within 30s |
| Sheets — Read | `testLoadFormData()` function | Form data logged to console |
| Sheets — Write | `testLogEntry()` function | New row appears in Agent Log |
| Gemini API | `testGeminiCall()` function | Valid JSON response returned |
| Gmail | `testSendEmail()` function | Test email received at alias |
| Chat webhook | `testChatWebhook()` function | Test message appears in Chat Space |
| Knowledge Base | `testLoadKnowledgeBase()` function | All keys return non-empty strings |

Run `runIntegrationHealthCheck()` in `Utils.gs` to execute all checks in sequence and log results.
